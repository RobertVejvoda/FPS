title: Authentication

Authentication is implemented using token-based methods to verify user identities. JSON Web Tokens (JWT) are used to securely transmit information between parties, ensuring data integrity and authenticity. OAuth 2.0 is employed to delegate access, allowing third-party services to interact with user data without exposing credentials. These methods provide a robust framework for managing user authentication and access control within the system.

### Token-Based Authentication

Token-based authentication is a protocol that allows users to verify their identity and receive a unique access token. This token is then used to access protected resources without needing to re-enter credentials. Commonly used tokens include [JSON Web Tokens (JWT)](https://jwt.io/introduction/).

#### JSON Web Tokens (JWT)

[JWT](https://jwt.io/introduction/) is a compact, URL-safe means of representing claims to be transferred between two parties. The token is digitally signed, ensuring its integrity and authenticity. A JWT consists of three parts: a header, a payload, and a signature.

- **Header**: Contains metadata about the token, such as the type of token and the hashing algorithm used.
- **Payload**: Contains the claims, which are statements about an entity (typically, the user) and additional data.
- **Signature**: Used to verify that the sender of the JWT is who it says it is and to ensure that the message wasn't changed along the way.

Tokens will be validated on the server side to ensure their authenticity and integrity.

#### Token Generation

Tokens are generated by [Azure API Management (APIM)](https://azure.microsoft.com/en-us/services/api-management/), which provides a secure and scalable way to manage and authenticate API requests. Azure APIM issues tokens that conform to industry standards, ensuring compatibility and security.

- **Azure APIM**: A comprehensive solution for managing APIs, including security, monitoring, and analytics. It supports token-based authentication methods such as JWT and OAuth 2.0.
- **Token Issuance**: Azure APIM can issue tokens based on various policies and configurations, allowing for flexible and secure authentication mechanisms.

#### Token Expiration

Token expiration is a critical aspect of token-based authentication, ensuring that tokens are valid only for a limited period. This reduces the risk of token misuse if a token is compromised.

- **Expiration Time**: Tokens should have a defined expiration time (`exp` claim in JWT), after which they are no longer valid. This time is typically set based on the security requirements of the application.
- **Refresh Tokens**: To maintain a seamless user experience, use refresh tokens to obtain new access tokens without requiring the user to re-authenticate. Refresh tokens should also have an expiration time and be securely stored.
- **Revocation**: Implement mechanisms to revoke tokens if necessary, such as when a user logs out or if suspicious activity is detected.

By managing token expiration effectively, you can enhance the security of your authentication system and protect user data.

#### Secure Storage for Tokens

Proper storage of tokens is crucial to maintaining the security of your authentication system. Tokens should be stored in a secure manner to prevent unauthorized access and misuse.

- **Client-Side Storage**: Store tokens in secure, client-side storage mechanisms such as HTTP-only cookies or secure storage APIs provided by the platform (e.g., `localStorage` or `sessionStorage` with appropriate security measures).
    - **HTTP-only Cookies**: These cookies are not accessible via JavaScript, reducing the risk of cross-site scripting (XSS) attacks.
    - **Secure Storage APIs**: Use platform-specific secure storage solutions, such as the Web Crypto API, to store tokens securely.

- **Server-Side Storage**: For server-side applications, store tokens in secure databases or in-memory storage with encryption to protect against unauthorized access.
    - **Encryption**: Encrypt tokens at rest and in transit to ensure they are protected from interception and unauthorized access.
        - **Access Controls**: Implement strict access controls to limit who and what can access the stored tokens. Refer to the [Access Control](./access-control) document for more details on managing access controls effectively.

- **Token Handling Best Practices**:
    - **Minimize Token Exposure**: Avoid exposing tokens in URLs or other insecure locations.
    - **Secure Transmission**: Always use HTTPS to transmit tokens securely over the network.
    - **Token Rotation**: Regularly rotate tokens to minimize the impact of a compromised token.

#### OAuth

[OAuth](https://oauth.net/) is an open standard for access delegation, commonly used as a way to grant websites or applications limited access to user information without exposing passwords. OAuth provides a secure and standardized method for token-based authentication and authorization.

- **OAuth 2.0**: The most widely used version, which introduces an authorization layer and separates the role of the client from that of the resource owner. It uses access tokens to allow third-party services to access user resources.

#### OAuth 2.0 in Web and Mobile Applications

OAuth 2.0 is implemented in both web and mobile applications to provide secure and seamless authentication and authorization experiences for users. By leveraging OAuth 2.0, applications can delegate access to user resources without exposing sensitive credentials.

In web and mobile applications, OAuth 2.0 is used to authenticate users and authorize access to APIs and other resources - authorization code flow with Proof Key for Code Exchange (PKCE) to enhance security.

- **Authorization Code Flow with PKCE**: This flow is designed for mobile and native applications, adding an extra layer of security by using a code verifier and code challenge.
    - **Step 1**: The app generates a code verifier and code challenge.
    - **Step 2**: The user is redirected to the authorization server to grant access.
    - **Step 3**: The authorization server redirects back to the app with an authorization code.
    - **Step 4**: The app exchanges the authorization code and code verifier for an access token.
    - **Step 5**: The access token is used to access protected resources.
